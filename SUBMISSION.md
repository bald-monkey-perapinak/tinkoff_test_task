# Submission Summary

## Что решает

Агент для поиска и предварительного отбора junior-вакансий/стажировок — автоматизирует ручной скролл hh.ru, фильтрует по критериям пользователя и выдаёт ранжированный markdown-отчёт с оценками соответствия и объяснениями. Поддерживает два режима: веб-интерфейс через Telegram Mini App и CLI без веб-стека.

## Как устроен pipeline

Публичный hh.ru API (без авторизации) → парсинг ответа (JSON/CSV) → LLM-анализ через Groq (Llama 3.3 70B) → markdown-отчёт. Параллельно работает Telegram Bot для уведомлений о новых вакансиях по подпискам. Весь pipeline повторяем и из CLI, и из веб-API — одни и те же функции `parse_uploaded_file`, `analyze_with_llm`, `generate_report`.

## Где LLM, а где обычная логика

LLM используется **только** в `analyze_with_llm` для ранжирования вакансий и генерации объяснений (почему подходит / что смущает). Всё остальное — детерминированная логика: парсинг файлов, генерация markdown-отчёта, поиск по hh.ru API, cron-проверка подписок, отправка уведомлений. При отсутствии Groq API-ключа работает rule-based fallback с тем же форматом вывода.

## Ограничения и риски

- **Groq API**: бесплатный tier с rate limit ~30 req/min; при превышении — автоматический retry с exponential backoff, затем rule-based fallback
- **hh.ru**: официальный публичный API без ключа; при сетевых блокировках IP — proxy-конфигурация для отказоустойчивости + полный mock-fallback на `data/sample_vacancies.json`
- **SQLite**: используется для избранного и подписок, подходит для прототипа, не для продакшена
- **Telegram Bot**: опциональный компонент, уведомления не критичны для основного функционала

## Что было сложнее всего

Балансировка между надёжностью интеграции с внешними сервисами (hh.ru, Groq) и простотой кода: retry-логика, proxy, mock-fallback, кэширование анализа по hash — всё добавлено, но код остаётся читаемым. А также sanitization промпта для LLM — защита от prompt injection без потери полезного контента (кириллица, спецсимволы зарплат).
